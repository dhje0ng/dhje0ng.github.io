function filterHtml($content) {
    $result = '';

    $html = new simple_html_dom();
    $html->load($content);
    $allowTag = ['a', 'img', 'p', 'span', 'br', 'hr', 'b', 'h1', 'h2', 'h3', 'h4', 'h5', 'h6', 'strong', 'em', 'code', 'iframe'];

    foreach($allowTag as $tag){
      foreach($html->find($tag) as $element) {
        switch ($tag) {
          case 'a':
            $result .= '<a href="' . str_replace('"', '', $element->href) . '">' . htmlspecialchars($element->innertext) . '</a>';
            break;
          case 'img':
            $result .= '<img src="' . str_replace('"', '', $element->src) . '">' . '</img>';
            break;
          case 'p':
          case 'span':
          case 'b':
          case 'h1':
          case 'h2':
          case 'h3':
          case 'h4':
          case 'h5':
          case 'h6':
          case 'strong':
          case 'em':
          case 'code':
            $result .= '<' . $tag . '>' . htmlspecialchars($element->innertext) . '</' . $tag . '>';
            break;
          case 'iframe':
            $src = $element->src;
            $host = parse_url($src)['host'];
            if (strpos($host, 'youtube.com') !== false){
              $result .= '<iframe src="'. str_replace('"', '', $src) .'"></iframe>';
            }
            break;
        }
      }
    }
    return $result;
  }

filterHtml() 함수가 있는데 분석해보니 조금 이상한 점을 발견할 수 있었다. case switch 문으로 들어가서

iframe 태그가 들어오면 parse_url() 함수에서 hostname 파싱해서 youtube.com 인지 검증하는 로직이 있다.

검증 로직을 pass하면 <iframe src=> 태그를 만들어서 $src 변수에 있는 값을 넣어서 return 해주게 된다.

그런데 parse_url() 함수를 사용할 때 schema 검사를 하지 않고 있어서 url schema를 자유롭게 사용할 수 있다.

즉 이 문제에서는 javascript:// 를 사용할 수가 있게 되는 것이다.

javascript:// 를 사용하면 parse_url() 함수에 의해 // 는 주석으로 취급되고, %0a 를 사용하면

개행으로 처리되서 %0a 뒤로 오는 코드는 새로운 javascript 코드로 실행되는 것이다.

즉 이 문제의 목표는 봇이 내 게시글을 방문하는데 여기서 XSS 통해서 내 서버로 Redirection 시키면

봇이 가지고 있는 정보(플래그)를 획득할 수 있다.

획득한 쿠키값 가지고 리얼 서버에서 PHPSESSID 세션 값 바꿔주면 플래그를 획득할 수 있다.

private static String lookupImg(String memo) {
    Pattern pattern = Pattern.compile("(\\[[^\\]]+\\])");
    Matcher matcher = pattern.matcher(memo);
    String img = "";
    if (!matcher.find()) {
        return "";
    }
    img = matcher.group();
    String tmp = img.substring(1, img.length() - 1);
    tmp = tmp.trim().toLowerCase();
    pattern = Pattern.compile("^[a-z]+:");
    matcher = pattern.matcher(tmp);
    if (!matcher.find() || matcher.group().startsWith("file")) {
        return "";
    }
    String urlContent = "";
    try {
        final URL url = new URL(tmp);
        final BufferedReader in = new BufferedReader(new InputStreamReader(url.openStream()));
        String inputLine = "";
        while ((inputLine = in.readLine()) != null) {
            urlContent = urlContent + inputLine + "\n";
        }
        in.close();
    }
    catch (Exception e) {
        return "";
    }
    final Base64.Encoder encoder = Base64.getEncoder();
    try {
        final String encodedString = new String(encoder.encode(urlContent.getBytes("utf-8")));
        memo = memo.replace(img, "<img src='data:image/jpeg;charset=utf-8;base64," + encodedString + "'><br/>");
        return memo;
    }
    catch (Exception e2) {
        return "";
    }
}

이 문제도 좀 아쉽게 못 푼 문제 중 하나인데, 결론은 Blind XPath Injection 기법을 통해 풀이가 가능했다.

private String[] doReadArticle(final HttpServletRequest req) {
    final String id = (String)req.getSession().getAttribute("id");
    final String idx = req.getParameter("idx");
    if ("null".equals(id) || idx == null) {
        return null;
    }
    final File userArticle = new File(this.tmpDir + "/article/", id + ".xml");
    try {
        final InputSource is = new InputSource(new FileInputStream(userArticle));
        final Document document = DocumentBuilderFactory.newInstance().newDocumentBuilder().parse(is);
        final XPath xpath = XPathFactory.newInstance().newXPath();
        String title = (String)xpath.evaluate("//article[@idx='" + idx + "']/title/text()", document, XPathConstants.STRING);
        String content = (String)xpath.evaluate("//article[@idx='" + idx + "']/content/text()", document, XPathConstants.STRING);
        title = this.decBase64(title.trim());
        content = this.decBase64(content.trim());
        return new String[] { title, content };
    }
    catch (Exception e) {
        System.out.println(e.getMessage());
        return null;
    }
}

ReadArticle() 함수를 보면 xpath 에 관한 함수를 사용하고 있다. 대표적으로 xpath.evaluate 인데

이 함수가 하는 역할은 인자로 들어가는 xpath 계산식을 실행하여 결과를 반환해준다.

xpath 에 연관이 높다고 생각해서 xpath injection 관련해서 많이 찾아보면서 여러가지 테스팅을 많이

시도해봤었는데 좀 식상한 개념이라 그런지 접근이 쫌 어려웠다.

일단 플래그 같은 경우 Dockerfile 에 보면 /usr/local/tomcat/conf/catalina.properties 파일에

정의되어 있었고 RCE 같은걸로 읽어내든 또는 injection 공격 통해서 플래그를 뽑아내든 해야 했었다.

결국 여러가지 삽질하면서 못 풀긴 했지만 대회 끝나고 공개된 풀이로는 catalina.properties 관련된 함수를

통해서 접근을 하신분이 계셨다.

https://developer.mozilla.org/en-US/docs/Web/XPath/Functions/system-property

대략적으로 system-property 는 시스템 속성을 나타내는 개체를 반환해주는데 xpath injection 취약점을

통해서 system-property 를 사용하면 개체를 불러올 수 있기 때문에 FLAG 값을 한글자씩 뽑아올 수 있게 된다.
